# Principles — Harmless Engineering 無害工程原則

> **每次程式碼變更都必須安全、可控、可回溯。不引入破壞性變更、不累積技術債、不犧牲品質換取速度。**

---

## Harmless Engineering 宣言

在 AI 輔助開發時代，程式碼生成速度大幅提升，但品質風險也同步增加。Harmless Engineering 是一套專為 AI 輔助開發設計的工程原則，確保：

| 原則 | 意涵 |
|------|------|
| 🛡️ **安全優先** | AI 生成的程式碼必須經過測試與審查，不盲目信任 |
| 🔄 **可回溯** | 每個 commit 都是最小可回溯單元，出問題能精確 revert |
| 📏 **小步迭代** | 每次只做 30-60 分鐘的原子任務，可驗證、可回溯 |
| 🚫 **零技術債** | 不允許「之後再修」的 hack，每個 commit 都是可部署的 |
| 🧪 **先測後寫** | 任何功能變更，測試先於實作 |
| 👁️ **人審 AI 出** | AI 是副駕駛（Co-Pilot），不是自動駕駛（Auto-Pilot） |

## 核心工程原則

### SOLID 原則

| 原則 | 說明 | AI 輔助開發中的應用 |
|------|------|---------------------|
| **S** 單一職責 | 一個模組只做一件事 | 要求 AI 拆分大函數為多個小函數 |
| **O** 開閉原則 | 對擴展開放，對修改關閉 | 新增功能用擴展而非修改既有程式碼 |
| **L** 里氏替換 | 子類別可替換父類別 | 檢查 AI 生成的繼承是否安全 |
| **I** 介面隔離 | 不強迫依賴不需要的介面 | 審查 AI 生成的介面是否過於臃腫 |
| **D** 依賴反轉 | 依賴抽象而非具體實作 | 確保 AI 生成的依賴是可替換的 |

### KISS / YAGNI / DRY

| 原則 | 全稱 | 說明 |
|------|------|------|
| **KISS** | Keep It Simple, Stupid | 簡單就是美。AI 容易過度工程化，必須要求它保持簡單 |
| **YAGNI** | You Aren't Gonna Need It | 不要預先實作「未來可能需要」的功能。只做當前需要的 |
| **DRY** | Don't Repeat Yourself | 不重複程式碼。AI 有時會複製貼上，必須檢查並重構 |

## AI 輔助開發安全守則

### 守則 1：AI 生成的程式碼必須通過測試

```
❌ 錯誤做法：
請 AI 生成 500 行程式碼 → 直接 commit

✅ 正確做法：
請 AI 先生成測試 → 審查測試是否合理
→ 請 AI 生成實作 → 跑測試 → 全部通過 → commit
```

### 守則 2：AI 生成的程式碼必須經過人工審查

審查重點：
- 邏輯是否正確？
- 邊界情況是否處理？
- 安全性是否有疑慮？
- 是否引入不必要的依賴？
- 是否符合專案的程式碼風格？

### 守則 3：任何變更不得破壞既有測試

```bash
# 每次 AI 生成程式碼後，必須執行
npm test        # 全部測試通過
npm run lint    # Lint 檢查通過
npm run typecheck  # 型別檢查通過（TypeScript）
```

### 守則 4：禁止「先 hack 再說」

```
❌ 「這個地方先用 hardcode，之後再改」
❌ 「這個錯誤先 catch 掉不處理，之後再補」
❌ 「這個函數先寫 300 行，之後再拆分」

✅ 每次 commit 都必須是可部署的品質
✅ 技術債在產生當下就必須解決
```

### 守則 5：每個 commit 都是可部署、可回溯的

```
✅ 好的 commit：
fix: 修復使用者登入後 token 未刷新的問題 (BUG-042)
feat: 新增郵件驗證流程，包含重送機制

❌ 不好的 commit：
WIP
fix stuff
update
```

### 守則 6：安全性相關程式碼須特別審查

以下類型的程式碼，AI 生成後必須人工逐行審查：
- 認證與授權邏輯
- 資料庫查詢（SQL injection 風險）
- 使用者輸入處理（XSS 風險）
- 密碼學相關程式碼
- API 金鑰或密鑰處理
- 檔案上傳處理

## 不可協商的編碼規則

### TypeScript / JavaScript 專案

```
1. 使用 TypeScript strict mode
2. 禁止使用 any 型別（除非有明確理由並加註解）
3. 所有函數必須有明確的 return type
4. 所有 API 呼叫必須有錯誤處理
5. 所有 async 函數必須有 try/catch
6. 禁止 console.log 留存於 production code（使用 logger）
```

### Python 專案

```
1. 使用 Type Hints
2. 遵循 PEP 8 風格
3. 所有公開函數必須有 docstring
4. 使用 mypy / pyright 進行靜態型別檢查
5. 所有外部 API 呼叫必須有錯誤處理與重試機制
```

### 通用規則

```
1. 所有環境變數與密鑰不得 hardcode
2. 所有錯誤訊息不得洩漏敏感資訊
3. 所有第三方依賴必須鎖定版本
4. 禁止提交含有敏感資訊的檔案（.env、憑證等）
```

## TDD：測試驅動開發循環

Harmless Engineering 的核心實踐是 TDD（Test-Driven Development）：

```
┌──────────────────────────────────────┐
│          RED → GREEN → REFACTOR       │
│                                      │
│   🔴 RED                             │
│   先寫一個會失敗的測試                 │
│   （定義「什麼是正確的行為」）           │
│         │                            │
│         ▼                            │
│   🟢 GREEN                           │
│   寫最少程式碼讓測試通過               │
│   （不追求優雅，只追求正確）             │
│         │                            │
│         ▼                            │
│   🔵 REFACTOR                        │
│   在測試的保護下重構程式碼              │
│   （提升可讀性、效能、結構）             │
│         │                            │
│         └────────→ 回到 🔴 RED       │
│                                      │
└──────────────────────────────────────┘
```

**在 AI 輔助開發中使用 TDD：**

```markdown
## 步驟 1：請 AI 生成測試
「請根據以下需求，產生測試案例：

需求：建立一個驗證 email 格式的函數 isValidEmail(email)

測試案例需涵蓋：
1. 正確的 email 格式
2. 缺少 @ 符號
3. 缺少域名
4. 空字串
5. 特殊字元
6. 邊界長度」

## 步驟 2：審查測試
確認測試案例是否合理、是否涵蓋邊界情況

## 步驟 3：請 AI 實作
「請實作 isValidEmail 函數，使其通過上述所有測試」

## 步驟 4：跑測試驗證
npm test -- --testPathPattern=isValidEmail
```

## 小步迭代：原子任務拆分

將大功能拆分為 30-60 分鐘可完成的原子任務：

```
❌ 錯誤任務：
「實作使用者管理系統」（太大，不可控）

✅ 正確拆分：
TASK-001: 建立 User model 與資料庫 schema     (30 min)
TASK-002: 實作 POST /api/users 註冊端點       (45 min)
TASK-003: 實作 GET /api/users/:id 查詢端點    (30 min)
TASK-004: 實作 PATCH /api/users/:id 更新端點  (30 min)
TASK-005: 實作 DELETE /api/users/:id 刪除端點 (30 min)
TASK-006: 加入使用者認證 middleware           (45 min)
TASK-007: 撰寫整合測試                        (30 min)
```

每個任務完成後：
1. 跑測試確認沒破壞既有功能
2. Commit 當前的變更
3. 更新 implementation-log.md
4. 繼續下一個任務

## AI 互動的安全檢查清單

每次 AI 生成程式碼後，執行以下檢查：

```
□ 程式碼是否通過所有測試？
□ 程式碼是否通過 lint 檢查？
□ 程式碼是否通過型別檢查（TypeScript/mypy）？
□ 是否有處理邊界情況（null、空值、異常輸入）？
□ 是否有處理錯誤情況（網路失敗、API 錯誤）？
□ 是否引入了新的依賴（需要審查安全性）？
□ 程式碼風格是否符合專案慣例？
□ 是否有 hardcode 的密鑰或敏感資訊？
□ commit message 是否清晰描述變更？
```

## 總結

Harmless Engineering 不是要減慢開發速度，而是要確保 AI 帶來的速度提升不會被技術債和 Bug 抵消。記住這六個字：

> **測、審、小、記、回、防**

| 字 | 含義 | 實踐 |
|----|------|------|
| 測 | 測試先行 | TDD 紅綠重構循環 |
| 審 | 人工審查 | AI 輸出必須審查 |
| 小 | 小步迭代 | 30-60 分鐘原子任務 |
| 記 | 記錄決策 | ADR + 實作日誌 |
| 回 | 可回溯 | 每個 commit 獨立可部署 |
| 防 | 防禦程式 | 邊界處理、錯誤處理 |
