## 基礎技能：描述需求

### 5W1H 需求框架

用 5W1H 精確描述需求，大幅提升 AI 輸出的品質：

| 問題 | 說明 | 範例 |
|------|------|------|
| **W**hat | 做什麼功能？ | 建立 email 驗證流程 |
| **W**hy | 為什麼需要？ | 防止機器人註冊，確保使用者是真實的 |
| **W**ho | 誰會使用？ | 新註冊使用者 |
| **W**hen | 何時觸發？ | 使用者提交註冊表單後 |
| **W**here | 在哪裡？ | 註冊頁面 → 收件匣 → 驗證頁面 |
| **H**ow | 如何實作？ | 發送含 token 的郵件，點擊連結完成驗證 |

```markdown
## 使用範例

What: 建立 email 驗證流程
Why: 防止機器人註冊，確保使用者 email 有效
Who: 新註冊使用者
When: 使用者提交註冊表單後
Where: POST /api/auth/register → 收件匣 → GET /api/auth/verify?token=xxx
How: 
- 註冊成功後產生 24 小時有效的驗證 token
- 發送含驗證連結的 email
- 使用者點擊連結 → 標記 email 為已驗證
- token 過期可重新發送
```

### 提供範例（輸入 → 期望輸出）

```markdown
## 範例

請建立一個函數 filterUsers，根據條件過濾使用者：

### 輸入
users = [
  { name: "Alice", age: 25, active: true },
  { name: "Bob", age: 17, active: false },
  { name: "Charlie", age: 30, active: true }
]
criteria = { age: ">= 18", active: true }

### 期望輸出
[
  { name: "Alice", age: 25, active: true },
  { name: "Charlie", age: 30, active: true }
]

### 要求
- 支援多條件組合（AND 邏輯）
- 支援比較運算子 >, <, >=, <=, ==, !=
- 如果沒有匹配結果，回傳空陣列
```

{{從 CONTEXT.md 貼上目錄結構}}

```

## 技能 3：Test Generation（生成測試）

**用途：** 請 AI 生成測試案例（TDD 的 Red 階段）

```markdown
請根據以下需求產生測試案例：

## 功能描述
{{描述要測試的功能}}

## 函數簽名
```{{語言}}
{{function signature}}
```

## 測試案例需涵蓋

### 正常情況
- {{正常輸入 → 期望輸出}}

### 邊界情況
- 空值、null、undefined
- 空陣列、空字串
- 極大值、極小值

### 錯誤情況
- 無效輸入
- 類型錯誤
- 超出範圍的輸入

### 回歸情況
- 參考 logs/bug-log.md 中相關的歷史 Bug

請使用 {{測試框架（如 Jest / Vitest / pytest）}} 語法產生測試程式碼。
每個測試案例請用清晰的描述命名（例如：`應該回傳 false 當輸入為空字串時`）。
```

### 針對既有程式碼補測試

```markdown
請為以下程式碼補上測試：

## 程式碼
```{{語言}}
{{貼上完整函數或模組}}
```

## 要求
- 測試覆蓋率目標：> 80%
- 優先測試核心邏輯路徑
- 包含邊界情況
- 使用 mock 隔離外部依賴
```

## 技能 5：Code Review（審查程式碼）

**用途：** 請 AI 審查程式碼

```markdown
請審查以下程式碼，重點關注：

## 功能正確性
- 邏輯是否正確？
- 是否滿足需求？

## 安全性
- SQL injection / XSS / CSRF 風險？
- 敏感資料是否妥善保護？
- 認證與授權是否正確？

## 錯誤處理
- 所有可能的例外是否處理？
- 錯誤訊息是否清晰但不洩漏敏感資訊？

## 效能
- N+1 query？
- 不必要的重複計算？
- 記憶體洩漏風險？

## 可維護性
- 是否符合 SOLID 原則？
- 命名是否清晰？
- 是否有重複程式碼？

## 程式碼
```{{語言}}
{{貼上待審查的程式碼}}
```

請列出你發現的問題，按嚴重程度排序，並提供修改建議。
```

## 技能 6：Refactoring（重構）

**用途：** 請 AI 協助重構程式碼

```markdown
請重構以下程式碼。

## 原始程式碼
```{{語言}}
{{貼上原始程式碼}}
```

## 重構目標
- {{例如：提取重複邏輯}}
- {{例如：拆分過大的函數}}
- {{例如：改善命名}}

## 限制
- 外部行為不能改變
- 所有既有測試必須繼續通過
- 保持相同的函數簽名（除非明確要求修改）

請重構程式碼，並說明：
1. 做了哪些變更
2. 為什麼這樣改
3. 哪些地方特別需要注意
```

## 技能 7：Explaining Code（解釋程式碼）

**用途：** 請 AI 解釋一段不熟悉的程式碼

```markdown
請解釋以下程式碼：

## 程式碼
```{{語言}}
{{貼上程式碼}}
```

## 我想了解
1. 這段程式碼的整體目的是什麼？
2. 每個關鍵部分的運作原理
3. 資料流的方向
4. 有什麼潛在問題或改進空間？

請用清晰的語言解釋，假設我對 {{相關技術}} 有基本的了解，但不熟悉這個特定的實作。
```

## 技能 8：Documentation Generation（產生文件）

**用途：** 請 AI 為函數/模組產生 API 文件

```markdown
請為以下程式碼產生文件：

## 程式碼
```{{語言}}
{{貼上函數或模組}}
```

## 文件格式要求
- 簡短的功能描述（一句話）
- 參數說明（名稱、型別、是否必填、說明）
- 回傳值說明（型別、說明）
- 使用範例（至少 2 個）
- 可能的錯誤與例外

請使用 {{JSDoc / docstring / OpenAPI}} 格式。
```

## 進階技能

### 小步迭代拆分

將大功能拆分為原子任務：

```markdown
請將以下功能拆分為可獨立實作的小任務：

## 功能
{{功能描述}}

## 拆分要求
- 每個任務 30-60 分鐘內可完成
- 每個任務可獨立測試
- 任務之間有明確的依賴關係
- 標註優先級（P0/P1/P2）

## 輸出格式
請用以下格式輸出：
- TASK-001 [P0]: {{任務標題}} ({{預估時間}})
  - 描述：{{做什麼}}
  - 依賴：{{依賴哪個 TASK}}
  - Acceptance Criteria：{{完成條件}}
```

- 優點：{{已知優點}}

## 常見陷阱與解法

### 陷阱 1：一次要求太多

```
❌ 「幫我建立整個使用者管理系統」

✅ 「請先建立 User model 與資料庫 migration」
   → 確認無誤後
   → 「接下來請建立 POST /api/users 註冊端點」
```

### 陷阱 2：上下文不足

```
❌ 直接丟程式碼給 AI，不解釋背景

✅ 使用 Context Loading pattern，先載入：
   - 專案願景
   - 技術棧
   - 程式碼風格
   - 相關的既有程式碼
```

### 陷阱 3：不審查直接使用

```
❌ AI 生成的程式碼直接 commit

✅ AI 生成 → 閱讀理解 → 跑測試 → 跑 lint 
   → 手動測試 → Code Review → commit
```

### 陷阱 4：對話斷層（沒有記憶）

```
❌ 每天開新對話從零開始

✅ 每次結束時更新 logs/implementation-log.md
   每次開始時複製 logs/ 內容給 AI
   AI 就能「記得」之前的決策
```

### 陷阱 5：模糊的需求描述

```
❌ 「幫我優化效能」

✅ 「使用者列表頁面在資料超過 1000 筆時載入需要 5 秒，
    請分析瓶頸並優化，目標是 2 秒內載入完成」
```

### 陷阱 6：忽略安全性

```
❌ 請 AI 生成認證邏輯後直接使用

✅ 要求 AI 解釋安全機制
   → 人工審查 token 處理、密碼儲存、session 管理
   → 確保符合 OWASP 最佳實踐
```

## 提示詞品質檢查清單

貼上 prompt 給 AI 前，檢查：

```
□ 是否說明了「做什麼」？（What）
□ 是否說明了「為什麼」？（Why）
□ 是否提供了足夠的上下文？
□ 是否提供了輸入/輸出的範例？
□ 是否說明了限制條件？
□ 是否拆分了任務（避免一次太多）？
□ 預期輸出格式是否清楚？
```

---

> **記住：** 好的 Prompt = 清晰的需求 + 足夠的上下文 + 明確的限制條件。Prompt 的品質直接決定 AI 輸出的品質。
