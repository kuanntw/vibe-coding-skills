# WORKFLOW.md — 開發工作流程

> **用途：** 定義所有開發場景的工作流程。每次開發時選擇對應的工作流程。
>
> **核心循環：** Context → Task → Implement → Test → Log → Commit

## 標準開發循環

**適用場景：** 開發新功能或進行重大變更

```
┌───────────────────────────────────────────────────────────┐
│                  THE VIBE CODING LOOP                     │
├───────────────────────────────────────────────────────────┤
│                                                           │
│   🌅 START SESSION                                        │
│      │                                                    │
│      ├──→ 1. 回顧日誌（5 分鐘）                             │
│      │                                                    │
│      ├──→ 2. 載入上下文到 AI                                │
│      │                                                    │
│      ├──→ 3. 從 dev-tasks.md 選擇任務                       │
│      │                                                    │
│      ├──→ 4. 與 AI 協作實作                                │
│      │       │                                            │
│      │       ├── 生成測試                                  │
│      │       ├── 實作功能                                  │
│      │       └── 優化調整                                  │
│      │                                                    │
│      ├──→ 5. 驗收（比對 acceptance criteria）              │
│      │                                                    │
│      ├──→ 6. 更新日誌                                     │
│      │                                                    │
│      └──→ 7. Commit & Push                                │
│                                                           │
│   🌙 END SESSION                                          │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

### Step 1：開始 Session — 回顧日誌（5 分鐘）

```markdown
□ 閱讀 logs/implementation-log.md 最近 3 筆記錄
□ 檢查 logs/bug-log.md 是否有需要優先處理的 Bug
□ 確認 dev-tasks.md 中今天的任務優先級
□ 注意任何阻擋或依賴項目
```

### Step 2：載入上下文到 AI

**最少上下文：**

```markdown
我正在開發 {{專案名稱}}。

## 專案願景
{{CONTEXT.md → 核心目標與邊界}}

## 目前狀態
{{CONTEXT.md → 技術棧、專案結構}}

## 今天的任務
{{dev-tasks.md → 具體任務}}
```

**當需要更多上下文時，追加：**

```markdown
## 技術設計
{{tech-design.md → 相關架構}}

## 近期變更
{{logs/implementation-log.md → 最近 1-2 筆}}

## 已知問題
{{logs/bug-log.md → 相關 Bug}}
```

### Step 3：選擇任務

**選擇優先級：**

| 優先級 | 條件 |
|--------|------|
| 🥇 第一 | 阻擋其他工作的任務 |
| 🥈 第二 | 影響使用者的 Bug |
| 🥉 第三 | 尚未完成的 P0 功能 |
| 4 | 尚未完成的 P1 功能 |
| 5 | 優化與打磨 |

### Step 4：與 AI 協作實作

```markdown
## 當前任務
{{dev-tasks.md → 任務描述}}

## 相關現有程式碼
{{貼上相關檔案內容或路徑}}

## 技術限制
{{tech-design.md → 架構限制}}

## 要求
請依據以上描述實作此任務，遵循技術設計中的模式。
實作後，請說明任何偏離設計的決策。
```

### Step 5：驗收

```
□ 程式碼可正常執行無錯誤
□ 核心功能正常運作
□ 邊界情況已處理
□ 無 console 警告或錯誤
□ 所有 acceptance criteria 已滿足
□ 跑過 npm test（全數通過）
□ 跑過 npm run lint（無錯誤）
```

### Step 6：更新日誌

在 `logs/implementation-log.md` 追加：

```markdown
### {{日期}} | Session: {{任務標題}}

**任務：** {{任務 ID}}
**耗時：** {{花費時間}}

#### 實作內容
- {{變更摘要}}

#### 修改的檔案
- `path/to/file.ts` — {{做了什麼}}
- `path/to/file2.ts` — {{做了什麼}}

#### 決策
- {{任何偏離原設計的決策與原因}}
```

### Step 7：Commit & Push

```bash
git add .
git commit -m "feat: {{簡短描述}}

Task: {{TASK-XXX}}

Changes:
- {{變更 1}}
- {{變更 2}}"
git push
```

## Bug 修復流程

```
┌───────────────────────────────────────────┐
│              BUG FIX WORKFLOW              │
├───────────────────────────────────────────┤
│                                           │
│   🐛 BUG 被發現                            │
│      │                                    │
│      ├──→ 1. 重現並記錄                     │
│      │                                    │
│      ├──→ 2. 調查根因                       │
│      │                                    │
│      ├──→ 3. 實作修復                       │
│      │                                    │
│      ├──→ 4. 測試修復 + 回歸測試             │
│      │                                    │
│      ├──→ 5. 加入預防（測試案例）             │
│      │                                    │
│      └──→ 6. 記錄並 commit                 │
│                                           │
│   ✅ BUG 關閉                              │
│                                           │
└───────────────────────────────────────────┘
```

### Step 1：重現並記錄 Bug

在 `logs/bug-log.md` 中新增：

```markdown
### BUG-{{編號}}: {{簡短描述}}

**狀態：** 🔴 待處理
**嚴重程度：** Critical | High | Medium | Low
**發現日期：** {{日期}}

#### 重現步驟
1. {{步驟 1}}
2. {{步驟 2}}
3. {{步驟 3}}

**預期行為：** {{預期}}
**實際行為：** {{實際}}

#### 錯誤訊息
{{貼上錯誤訊息或 stack trace}}

#### 環境
- 瀏覽器/裝置：{{資訊}}
- 版本：{{版本號}}
```

### Step 2：與 AI 合作調查根因

```markdown
我需要幫助調查這個 Bug。

## Bug 描述
{{bug-log.md → Bug 記錄}}

## 相關程式碼
{{貼上相關檔案內容}}

## 我的觀察
{{已經發現的線索}}

請幫助我找出根本原因。
```

### Step 3：實作修復

```markdown
根因是 {{根因}}。

請實作修復方案：
1. 解決這個問題
2. 不破壞既有功能
3. 最小化變更範圍

請說明修改內容與原因。
```

### Step 4：測試修復與回歸

```
□ 原始重現步驟不再觸發 Bug
□ 相關功能仍正常運作
□ 無新增 console 錯誤
□ 所有既有測試通過
```

### Step 5：加入預防（回歸測試）

```markdown
請幫我撰寫一個測試案例，防止此 Bug 再次出現。

## Bug 描述
{{bug-log.md → Bug 記錄}}

## 修復內容
{{程式碼變更}}

請建立測試案例：
1. 應該在此 Bug 被引入時就偵測到
2. 驗證修復方案有效
3. 防止未來回歸
```

### Step 6：記錄並 Commit

更新 `logs/bug-log.md`：

```markdown
**狀態：** ✅ 已修復
**修復日期：** {{日期}}

#### 根因
{{根本原因}}

#### 修復方案
{{修復描述}}

#### 預防措施
- 新增測試：{{測試檔案}}
```

Commit：

```bash
git commit -m "fix: {{描述}} (BUG-{{編號}})

Root cause: {{根因}}
Prevention: {{測試名稱}}"
```

## 重構流程

```
┌───────────────────────────────────────────┐
│            REFACTORING WORKFLOW            │
├───────────────────────────────────────────┤
│                                           │
│   📝 記錄重構意圖                           │
│      │                                    │
│      ├──→ 1. 建立 ADR（為什麼重構？）        │
│      │                                    │
│      ├──→ 2. 跑所有測試（建立基線）           │
│      │                                    │
│      ├──→ 3. 規劃小步驟                      │
│      │                                    │
│      ├──→ 4. 逐步執行                       │
│      │       │                            │
│      │       ├── 重構                       │
│      │       ├── 測試                       │
│      │       ├── Commit                    │
│      │       └── 重複                       │
│      │                                    │
│      └──→ 5. 更新文件                       │
│                                           │
│   ✨ 重構完成                              │
│                                           │
└───────────────────────────────────────────┘
```

### Step 1：建立 ADR（架構決策記錄）

在 `logs/decisions-log.md` 新增：

```markdown
### ADR-{{編號}}: 重構 {{目標}}

**狀態：** 進行中
**日期：** {{日期}}

#### 背景
{{為什麼需要重構？}}

#### 當前問題
{{現在架構的問題}}

#### 目標狀態
{{期望的結果}}

#### 方案
{{高層次的拆分計畫}}
```

### Step 2：建立測試基線

```bash
# 跑所有測試，確保重構前一切正常
npm test

# 記錄結果：Tests: XX passed, 0 failed
```

### Step 3：規劃小步驟

每個步驟應：
- 可在 15-30 分鐘內完成
- 不改變外部行為
- 可獨立測試
- 可獨立 revert

```markdown
□ Step 1: 提取函數 X 到獨立檔案
□ Step 2: 重新命名變數提升可讀性
□ Step 3: 抽取共用邏輯
□ Step 4: 更新 import 路徑
□ Step 5: 移除死碼
```

### Step 4：逐步執行

每一步完成後：

```bash
npm test          # 測試必須仍然全過
git commit -m "refactor: {{具體變更內容}}"
```

與 AI 協作重構的 prompt：

```markdown
我正在重構 {{元件}}。

## 當前程式碼
{{貼上程式碼}}

## 這一步的目標
{{具體的重構目標}}

## 限制
外部行為不能改變。
請進行重構並說明變更內容。
```

### Step 5：更新文件

```
□ 如果架構變更 → 更新 CONTEXT.md
□ 如果模式改變 → 更新相關 tech-design.md
□ 更新 ADR 狀態為「已完成」
□ 記錄學到的經驗到 logs/insights.md
```

## 快速修復流程

**適用場景：** 小範圍變更（< 5 個檔案，< 30 分鐘）

```
┌───────────────────────────────────────────┐
│            QUICK FIX WORKFLOW              │
├───────────────────────────────────────────┤
│                                           │
│   ⚡ 快速變更                              │
│      │                                    │
│      ├──→ 1. 確認範圍（< 5 檔案）           │
│      │                                    │
│      ├──→ 2. AI 輔助實作                    │
│      │                                    │
│      ├──→ 3. 快速驗證                       │
│      │                                    │
│      └──→ 4. Commit                       │
│                                           │
│   ✅ 完成                                  │
│                                           │
└───────────────────────────────────────────┘
```

**使用時機：**
- 修正錯字
- 小範圍樣式調整
- 設定檔更新
- 小範圍文字變更
- 補上缺少的屬性

**Prompt 模式：**

```markdown
快速變更需求：

## 要做什麼

## 目標檔案
{{檔案路徑}}

## 當前內容
{{當前程式碼片段}}

```

**Commit：**

```bash
git commit -m "fix: {{簡短描述}}"
```

## 程式碼審查流程

```
┌───────────────────────────────────────────┐
│            CODE REVIEW WORKFLOW            │
├───────────────────────────────────────────┤
│                                           │
│   📝 待審查程式碼                            │
│      │                                    │
│      ├──→ 1. 理解變更意圖                     │
│      │                                    │
│      ├──→ 2. AI 輔助審查                     │
│      │                                    │
│      ├──→ 3. 人工驗證                        │
│      │                                    │
│      ├──→ 4. 提供回饋                        │
│      │                                    │
│      └──→ 5. 核准或要求修改                   │
│                                           │
│   ✅ 合併                                  │
│                                           │
└───────────────────────────────────────────┘
```

### AI 輔助審查 Prompt

```markdown
請審查以下程式碼，重點關注：

## 功能正確性
- 邏輯是否正確？
- 邊界情況是否處理？

## 安全性
- 是否有注入風險？
- 敏感資料是否妥善處理？

## 效能
- 是否有不必要的重複計算？
- 是否有記憶體洩漏風險？

## 可維護性
- 程式碼是否易於理解？
- 是否符合 SOLID 原則？

## 程式碼
{{貼上待審查的 diff 或完整檔案}}
```

### 審查檢查清單

```
□ 邏輯正確，無明顯 Bug
□ 邊界情況有處理（null、空值、異常輸入）
□ 錯誤情況有處理（網路失敗、API 錯誤）
□ 無安全性漏洞（注入、XSS、權限問題）
□ 無效能問題（N+1 query、不必要的重複計算）
□ 符合專案程式碼風格（命名、格式）
□ 有足夠的測試覆蓋
□ Commit message 清晰
□ 無引入不必要的依賴
```

## Commit Message 規範

### 格式

```
<type>: <簡短描述>

[可選的詳細說明]

[參考資訊，如 TASK-XXX、BUG-XXX]
```

### Type 類型

| Type | 用途 | 範例 |
|------|------|------|
| `feat` | 新功能 | `feat: 新增使用者 email 驗證流程` |
| `fix` | Bug 修復 | `fix: 修復 token 過期未自動刷新 (BUG-042)` |
| `refactor` | 重構（不改變行為） | `refactor: 提取共用驗證邏輯至 validator.ts` |
| `test` | 新增或修改測試 | `test: 新增 email 驗證邊界測試` |
| `docs` | 文件更新 | `docs: 更新 API 文件中的認證說明` |
| `style` | 格式調整 | `style: 統一 import 排序` |
| `chore` | 維護工作 | `chore: 更新依賴版本` |
| `perf` | 效能優化 | `perf: 優化使用者列表查詢的 N+1 問題` |

## 每日啟動檢查清單

```markdown
## 早上 5 分鐘
□ 回顧昨天最後的 commit
□ 閱讀 logs/implementation-log.md 最新記錄
□ 確認今天的優先任務

## 開始寫程式前
□ 複製 CONTEXT.md 相關內容
□ 貼到 AI 對話開頭
□ 確認今天要用的 Workflow

## 結束時
□ 更新 logs/implementation-log.md
□ 更新 logs/decisions-log.md（如果有重要決策）
□ Commit 所有變更
□ Push 到 remote
```
